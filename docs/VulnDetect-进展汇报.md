# VulnDetect 项目进展汇报

> 汇报日期：2026-07-02 | 版本 1.0

---

## 一、项目概况

VulnDetect 是一个**漏洞检测领域 LLM 训练与验证框架**，目标是让开源小模型（Qwen-3B）通过微调获得代码安全审计能力，能够分析代码、发现漏洞、给出修复建议。

项目代号 "ndfactory"（Next-generation Defect Factory），寓意将安全专家的缺陷检测经验工厂化、自动化。

**项目仓库**：https://github.com/SlainTroyard/ndfactory

---

## 二、已完成工作

### 2.1 平台建设总览

经过前期开发，我们已建成一条**从数据到模型的完整工具链**，包含 5 个子系统：

```
数据采集 ──→ 模型训练 ──→ 自动评测 ──→ Web 服务 ──→ 前端展示
 (完成)       (完成)       (完成)       (完成)       (完成)
```

### 2.2 各模块详情

#### 数据管线

- 实现 NVD（美国国家漏洞数据库）API 自动采集，支持分页和严重度过滤
- 实现 GitHub Security Advisory GraphQL 采集
- 内置去重、文本规范化、严重度分级过滤
- 自动格式化为标准的对话训练数据格式（OpenRLHF）
- **当前积累**：447 条漏洞描述数据（402 训练 / 45 验证）

#### 训练管线

- 基于 Qwen2.5-3B-Instruct 基座模型
- 实现 QLoRA 高效微调（4-bit 量化 + LoRA 低秩适配），单卡 A6000 (48GB) 即可训练
- 支持 **SFT → DPO → PPO** 三阶段训练链路
  - SFT（监督微调）：基于 HuggingFace Trainer
  - DPO（直接偏好优化）：**纯 PyTorch 自研实现**，不依赖第三方 RLHF 库
  - PPO（近端策略优化）：骨架已搭建
- 集成 DeepSpeed ZeRO-2 分布式训练加速
- 训练日志实时写入数据库，支持前端监控

#### 评测管线

- 基于 lm-evaluation-harness 的自动化评测框架
- 覆盖 5 个安全领域 benchmark：
  - MMLU 计算机安全子集（综合知识）
  - VulnBench（漏洞检测专项）
  - SecEval（9 领域安全评估）
  - CyberMetric（网络安全度量）
  - CTIBench（威胁情报推理）
- 支持**基座模型 / SFT 模型 / DPO 模型**三路对比评测
- 评测结果自动解析、入库、可追溯

#### Web 后端

- 基于 FastAPI + SQLAlchemy + SQLite
- 提供实验管理 API（创建/启动/暂停/查询）
- 提供模型推理 API（动态加载 QLoRA 适配器，支持对话）
- WebSocket 实时推送训练指标（loss、学习率、GPU 显存）
- 数据库设计 5 张表：Experiment、Checkpoint、TrainingMetric、Evaluation、Dataset

#### Web 前端

- 基于 React 18 + TypeScript + Vite + Tailwind CSS
- GitHub Dark 专业风格界面
- 5 个功能页面：
  1. **Dashboard**：实验总览、统计卡片、自动刷新
  2. **训练监控**：双曲线（SFT/DPO）同图对比、实时 loss 曲线
  3. **评测报告**：多实验分数对比表、颜色标注
  4. **数据管理**：数据集元信息展示
  5. **Playground**：Checkpoint 动态加载、交互式漏洞检测对话
- 支持 3-5 秒自动轮询刷新

#### 自动化工具

- `run_all.sh`：一键运行完整流程（数据清洗 → SFT → DPO → 评测 → DB）
- `setup_env.sh`：一键环境搭建（GPU 检测 + venv + PyTorch CUDA 12.1）
- `reset.sh`：快速重置实验环境

### 2.4 从零到可演示的完整流程

```bash
# 一行命令跑通全链路
bash scripts/run_all.sh

# 启动 Web 界面
uvicorn vulndetect.backend.main:app --host 0.0.0.0 --port 8000
cd vulndetect/frontend && npm run dev
```

整个过程自动化，从数据采集到模型训练到 Web 演示，无需人工干预。

## 三、现阶段评估与问题发现

### 3.1 平台层面：已就绪

工具链完整、可用、可复现。一个新人拿到仓库后，按操作手册 30 分钟内可以跑通全流程。

### 3.2 模型层面：存在根本性问题

经过深入分析，我们发现**当前模型的训练数据存在严重缺陷**。

**问题根源**：训练数据来自 NVD CVE 描述 → 格式化为对话。实际抽查的数据样本如下：

```json
{
  "human": "CVE ID: CVE-2018-25380\nSeverity: HIGH\nDescription: Joomla Component eXtroForms 2.1.5 contains an SQL injection vulnerability...",
  "gpt": "This vulnerability (CVE-2018-25380) has a severity of HIGH.\nNo specific fix provided."
}
```

**关键问题**：

| 问题 | 影响 |
|------|------|
| 模型学到的输出是"复述 CVE 信息 + severity HIGH" | 不具备漏洞检测推理能力 |
| 训练数据中没有代码、没有推理过程、没有修复方案 | 模型无法分析输入代码 |
| 数据本质是"CVE 字典背诵"而非"安全审计方法" | 对未见过的新代码完全无效 |
| 447 条数据全是此模式 | 泛化能力几乎为零 |

换句话说：**平台是好的，但当前这条数据策略训练出来的模型，能力上限极低。** 它只是在背漏洞百科全书，而不是在学习如何发现漏洞。

### 3.3 方向确认

我们不是在否定已完成的工作。恰恰相反——完整的工具链为后续迭代提供了坚实底座。现在的问题聚焦在一点：**训练数据怎么做，模型才能真正学会漏洞检测？**

## 四、技术调研：业界实践启示

在发现上述问题后，我们调研了业界前沿的个人开发者实践。

### 4.1 关键案例：yuxinlu1 的蒸馏路线

2026 年 6 月，一位个人开发者（逯雨鑫，美国 AI 方向研究生）的模型连续多日霸榜 HuggingFace Trending，下载量超 70 万，力压智谱 GLM-5.2、百度 Unlimited-OCR、Kimi K2.7 等大厂模型。

他的核心做法：

1. **用顶级大模型（Claude Opus / Fable 5）作为教师**，生成高质量编程推理数据
2. **蒸馏到 12B 小模型**（Gemma-4-12B），QLoRA 微调，仅 ~1 万条数据
3. **量化部署**（GGUF），最低 4.5GB 显存可跑，本地离线零成本
4. **数据验证闭环**：代码推理链必须真跑过测试、通过了才保留

结果：在 tau2-bench telecom 基准上，基座 15% → 蒸馏后 55%，提升 3.5 倍。

硬件：单张 RTX 5090 (32GB)，总投入约 40 小时，纯自费。

### 4.2 核心启示

| 启示 | 对我们的意义 |
|------|-------------|
| **数据质量 > 数据数量** | 我们不需要海量数据，~500 条高质量推理链就可能显著提升 |
| **蒸馏是个人开发者可行的路线** | 不需要大厂算力，单卡 A6000 (48GB) 比他的 5090 更充裕 |
| **专注一个痛点** | 他做通用编程，我们聚焦漏洞检测——更垂直、评判标准更明确 |
| **部署门槛决定影响力** | 3B 模型量化后可在 8GB 统一内存设备上运行 |

## 五、下一步计划：蒸馏实验

### 5.1 核心思路

**不变**：现有的平台工具链（数据管线、训练管线、评测管线、Web 前后端）

**改变**：训练数据的生成策略

```
当前方案：NVD API → CVE 描述 → 格式化为对话 → SFT 训练
    ↓ （模型学会了背 CVE 字典）

新方案：真实漏洞代码 → Claude Opus 安全审计 → 推理链数据 → 蒸馏到 3B 模型
    ↓ （模型学会了像安全专家一样分析代码）
```

### 5.2 实验设计：三组对照

| 组别 | 训练数据 | 目的 |
|------|---------|------|
| **A（对照组）** | 无（基座模型原始能力） | 能力天花板基准 |
| **B（当前方案）** | NVD CVE 描述数据 | 当前方案的实际效果量化 |
| **C（新方案）** | 教师模型推理链数据 | 验证蒸馏是否有效 |

三组使用**完全相同的**基座模型、训练参数、评测标准，唯一变量是训练数据来源。

### 5.3 执行计划（6 个阶段，预计 4-5 天）

| 阶段 | 内容 | 耗时 | 关键交付 |
|------|------|------|---------|
| P0 | 环境准备 + 构造评测集 | 2h | API 配置、100 条 hold-out 评测样本 |
| P1 | 收集代码样本（GitHub Advisory + NVD diff + SARD + OWASP） | 1d | 600+ 真实漏洞代码样本 |
| P2 | Claude Opus 生成推理链数据（Pilot → 批量 → 校验） | 1d | 400+ 条高质量训练数据 |
| P3 | 训练基线模型 + 评测（组 A + 组 B） | 0.5d | 当前方案的量化基线 |
| P4 | 训练蒸馏模型 + 评测（组 C） | 0.5d | 新方案的实际效果 |
| P5 | 三组对比分析 + 实验报告 + 决定下一步 | 0.5d | 明确结论与决策 |

**P1 与 P3 可并行执行，关键路径约 3.5 天。**

### 5.4 预期效果

| 指标 | 当前方案（B） | 蒸馏方案（C，预期） | 提升 |
|------|-------------|-------------------|------|
| 通用安全 benchmark | 基座 + 5~10 分 | 基座 + 3~8 分 | 持平或微降 |
| 代码漏洞检测 F1 | ~15%（几乎随机） | **~55%** | **+40 个百分点** |
| 漏洞类型归类准确度 | ~20% | ~60% | +40 个百分点 |
| 修复方案可用性 | 0%（不输出修复） | ~50% | 从无到有 |

通用能力不降、专项能力大幅提升是我们的设计目标。

### 5.5 风险评估

| 风险 | 概率 | 应对 |
|------|------|------|
| 蒸馏后通用能力下降（灾难性遗忘） | 中 | 5 个 benchmark 全程监控，必要时混合通用数据 |
| 教师模型幻觉（编造漏洞） | 中 | 自动校验 + 人工 10% 抽查 |
| API 成本超预期 | 低 | 先 50 条 pilot 验证再扩量，总预算控制 $100 内 |
| 实验无效果 | 低 | 若失败则意味着 3B 模型容量不足，需考虑更大底座 |

### 5.6 资源需求

| 资源 | 需求 | 现状 |
|------|------|------|
| GPU | A6000 48GB × 1 | ✅ 已就绪 |
| 教师模型 API | Claude Opus 4.8 | ✅ 已有 Max 套餐 |
| API 预算 | ~$65（pilot + 正式） | ✅ 已有额度 |
| 人力 | 1 人，4-5 天 | - |

## 六、中长期展望

### 6.1 实验成功路线（如果 Phase 5 结论为 ✅）

```
Phase 5 成功 → DPO 对齐降误报 → GGUF 量化部署
    → HuggingFace 发布（效仿 yuxinlu1）
    → 社区反馈迭代 → 扩展到 7B/12B 底座
```

### 6.2 实验失败路线（如果 Phase 5 结论为 ❌）

```
Phase 5 失败 → 分析失败维度 → 调整数据策略重新实验
    → 或考虑更大的底座模型（7B+）
    → 或与安全公司合作获取更多真实漏洞数据
```

### 6.3 长期愿景

构建一个**本地、离线、低成本**的代码安全审计助手：

- 开发者提交代码 → 本地模型自动审计 → 发现漏洞 + 给出修复建议
- 不需要把代码上传到云端（保护隐私）
- 不需要为每次审计付费（降低门槛）
- 覆盖 OWASP Top 10 等常见漏洞类型

## 七、总结

### 已完成

- ✅ 完整的 LLM 微调平台（数据 → 训练 → 评测 → Web 前后端）
- ✅ 一键自动化流程（`bash scripts/run_all.sh`）
- ✅ QLoRA 三阶段训练链路（SFT + 自研 DPO + PPO 骨架）
- ✅ 5 个安全 benchmark 的自动化评测体系
- ✅ GitHub Dark 风格的前端 Dashboard（5 个页面）

### 待推进

- 🔄 训练数据策略升级：从"CVE 描述 → SFT"到"教师推理链 → 蒸馏"
- 🔄 蒸馏实验验证（4-5 天，~$65 API 成本）
- 🔄 根据实验结果决定下一步方向

### 一句话总结

**平台已经准备好了。现在需要一个更好的数据策略，让模型从"背字典"进化为"会思考"。蒸馏路线是个人开发者验证过的可行路径，我们只需要在漏洞检测这个垂直领域复现这个逻辑。**
